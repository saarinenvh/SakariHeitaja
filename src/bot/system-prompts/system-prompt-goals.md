# SAKKE_SYSTEM_PROMPT.md

## Goal

We should step back from prompt-tuning and redesign the AI integration around clear responsibilities.

The current procedural commentary is already good at:
- being funny in the existing Sakke style
- staying factually correct
- producing predictable output

The AI integration should improve three things:

1. **Competition context awareness**
	- The commentary should understand not only the hole result, but also what it means in the competition.
	- Example: moved into the lead, dropped from the lead, stayed in contention, late-round pressure, etc.

2. **More fluent language**
	- The output should sound more natural and flow better than rigid procedural sentences.
	- It should still preserve the same rough, funny Sakke energy.

3. **More variety**
	- The current phrase pools are funny, but everyone in the club already knows the lines.
	- We want more random commentary, more new punchlines, and more variation, while still keeping the same recognizable personality.

---

## Important Design Principle

Do **not** use the LLM to edit an already-written full procedural comment.

That path became awkward and unstable.

Instead, the better model is:

- **Code decides what happened**
- **Code decides why it matters**
- **LLM decides how to say it**

So the LLM should not receive a finished commentary line to "polish".
It should receive a **structured commentary brief** generated by code.

---

## Responsibility Split

### What code should do

Code should remain responsible for all deterministic logic:

- score parsing
- change detection
- score type classification
- leaderboard changes
- competition state tracking
- identifying whether the event matters
- deciding whether commentary should be generated
- deciding whether the event is:
	- good
	- neutral
	- bad
- deciding whether competition context is significant
- choosing optional style hints or legacy flavor hints
- appending final structured data like score / placement if needed

In short:

> Code understands the event and competition state.

### What the system prompt + LLM should do

The LLM should be responsible for:

- turning a structured brief into natural commentary
- making the language flow better
- adding humor and variation
- reacting to context in Sakke style
- occasionally producing new funny throw descriptions or punchlines

In short:

> The LLM writes the commentary, but only from the facts the code already prepared.

---

## Recommended Architecture

### Current bad abstraction to avoid

Avoid this model:

```text
procedural full comment
→ LLM rewrites it

This tends to produce:

weird rewrites
loss of structure
loss of facts
random append-only behavior
awkward editing behavior that LLMs are not naturally good at
Better abstraction

Use this:

Metrix event + competition state
→ code builds CommentaryBrief
→ LLM generates short Sakke commentary
→ code appends structured tail if needed
What CommentaryBrief should contain

Create a new structured object for commentary generation.

Suggested shape:

type CommentaryBrief = {
	playerName: string;
	holeNumber: number;
	holeScoreLabel: string; // e.g. "birdie", "bogey", "par", "eagle", "double bogey"
	eventQuality: "good" | "neutral" | "bad";
	ob: boolean;
	positionChange?: "moved_up" | "moved_down" | "no_change";
	standingImpact?: string; // e.g. "moved into solo lead", "moved into shared lead", "dropped from lead", "still leading", "close behind leader"
	competitionPhase?: "early" | "mid" | "late";
	pressureMoment: boolean;
	styleMode?: "safe" | "chaos";
	reactionHints?: string[];
	scoreSlangHints?: string[];
	verbHints?: string[];
	descriptionHints?: string[];
};

This object should be produced entirely in code.

Why this is better

This solves the three main goals directly:

1. Competition context awareness

The code can explicitly calculate things like:

moved into solo lead
tied for the lead
dropped out of lead
now one shot behind leader
late-round pressure moment

The LLM does not need to infer this from raw leaderboard data.

2. More fluent language

The LLM is better at writing from structured facts than editing finished text.

3. More variety

The LLM can generate fresh phrasing, while code can optionally provide small legacy Sakke hints for style.

What the code should calculate before calling the LLM

The LLM should not receive raw Metrix state if avoidable.

Instead, code should precompute commentary-relevant facts such as:

player name
hole result label
whether OB happened
whether the result was good / neutral / bad
whether the player moved up or down
whether the player is now leading / tied for lead / close behind
whether this happened early, mid, or late in the round
whether the moment is high pressure

Code should reduce all of this into a simple brief.

This is important because:

LLMs are good at expression, not reliable competition reasoning.

Suggested competition context tags

Instead of feeding raw scoreboard details, generate short tags / summaries like:

"moved into solo lead"
"moved into shared lead"
"dropped from solo lead"
"dropped from lead"
"still leading despite the mistake"
"still close behind the leader"
"gained ground on the leaders"
"no major standing impact"
"late round pressure moment"
"must-finish strong situation"

These can then be passed into the prompt.

Suggested style hints from legacy Sakke system

The current procedural system already contains good flavor DNA:

startTexts
scoreTexts
verbs
narrative style
absurd throw descriptions

Do not throw that away.

But do not dump the full lists every time either.

Instead, code can inject a small random sample into the brief or user prompt.

For example:

2 reaction hints
2 score slang hints
2 verb hints
2 absurd description hints

This preserves Sakke’s recognizable style without making all outputs identical.

Example:

reactionHints: ["WOUUUU", "Voi vittu"]
scoreSlangHints: ["pörö", "ruskea boggeli"]
verbHints: ["paukutti", "möyri"]
descriptionHints: ["Iha siilon yli ja sit suoraan kuoppaan, perkele!", "Keskellä lätäkköä, toivottavasti on sukelluskamat messissä"]
Recommended generation modes

Add a simple mode field in code:

Safe mode

Use when:

competition context matters
event is important
we want reliable output

Characteristics:

cleaner output
less chaotic
more focus on the competition situation
Chaos mode

Use when:

lower-stakes event
casual commentary
we want more absurd Sakke flavor

Characteristics:

more playful
more random punchlines
more exaggerated throw descriptions

This gives better pacing and avoids every comment feeling equally over-the-top.

Recommended output shape

Do not force every single comment into the exact same structure.

Instead:

Default: 2 sentences
Use 3 sentences only when competition context is meaningful

Suggested logic:

For ordinary events

2 sentences:

reaction + player + result
humorous description / reaction
For important competition events

3 sentences:

reaction + player + result
humorous throw description
short reaction to competition situation

This allows competition context to appear naturally when it actually matters.

Suggested implementation plan
Step 1 — Build CommentaryBrief in code

Add a dedicated builder function that converts:

change
competitionName
results
state / round context

into a compact CommentaryBrief.

Do this first without changing the current output behavior.

Step 2 — Define competition context helpers

Create helper functions for:

standing impact
competition phase
pressure moment
position movement
relevance level

These should remain fully deterministic.

Step 3 — Build a new LLM prompt around the brief

Do not pass finished commentary text.
Pass a structured summary generated from CommentaryBrief.

The prompt should instruct the model to:

write natural Finnish spoken-language commentary
mention competition context if present
stay short
stay in Sakke tone
avoid inventing facts
Step 4 — Keep procedural commentary as fallback

If LLM fails or output is invalid:

use the old procedural generator

This gives safe rollout.

Step 5 — Add optional style hint sampling

From existing legacy pools, add small random samples to improve continuity of style.

Step 6 — A/B test

Compare:

old procedural commentary
new brief-based AI commentary

Evaluate:

factual correctness
competition awareness
naturalness
humor
variation

Validation and safety

Even with brief-based prompting, the code should still validate output.

Recommended checks:

includes the correct player name
does not include invented other player names
does not include forbidden numbers if they are supposed to be appended separately
is not excessively long
is not empty
does not include obvious hallucinated competition facts

If validation fails:

use procedural fallback
Key insight

The correct abstraction is not:

LLM edits existing text

The correct abstraction is:

Code builds meaning
LLM builds language

This keeps:

reliability in code
natural language in AI
competition understanding in deterministic logic
Sakke flavor in prompts + optional style hint pools
Final recommendation

The new AI integration should be built around a CommentaryBrief.

That gives us the best path toward the three goals:

Competition context awareness
achieved by code precomputing standing / pressure / phase
More fluent language
achieved by generating commentary from structured facts instead of rewriting procedural text
More random and fresh commentary
achieved by LLM generation plus optional sampled legacy Sakke hints

This is the cleanest long-term architecture and avoids getting trapped in endless prompt-tuning around rewriting already-finished lines.
